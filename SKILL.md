# 学习日记

## 概述

从与 AI 的交互中提取知识，以老师的口吻编写学习文档。不写练习题，只讲知识。**讲知识要讲透，带实操细节、可复用的知识、以及 AI 的思考过程**，让学生不仅学到"做了什么"，更能学到"怎么想到的"。

**核心原则：** 像老师手把手教学一样——讲清楚原理，给出操作步骤和代码，还要把思考过程摊开来给你看。

## 何时使用

- 对话上下文即将被压缩（信息即将丢失）
- 人类伙伴调用 `/learning-journal` 或要求捕获学习内容
- 对话中涉及了新概念、新工具、新原理
- 讨论中解释了某个技术原理或工作机制
- 对话中有丰富的知识探索过程

## 知识提取流程

### 第一步：扫描对话中的知识点

回顾完整对话历史，识别所有涉及的知识和思考过程：

- 讨论或解释了什么概念和原理
- 使用了什么工具、API、库、框架（涉及什么知识才能用）
- 触及了什么技术领域的基础理论
- 从对话中延伸出了什么关联知识
- AI 补充了什么背景知识
- 这些知识会在什么情况下可能会用到
- 对用户多次提到的问题进行更加详细的教学
- **AI 的思考过程** — AI 是怎么一步步推理到结论的？看到了什么线索？排除了什么错误方向？为什么选择了这条路径而不是另一条？

### 第二步：按知识领域归类

将散落的知识点按领域分组，而不是按问题分组。

### 第三步：以老师的口吻讲知识

**老师的口吻意味着：**

- **直接讲解** — "PE 文件的结构是这样的..."，而不是"PE 文件格式是...它很重要因为..."
- **由浅入深** — 先给直觉和宏观图景，再深入细节
- **有实操** — 讲到工具怎么用，要给出具体的命令、参数、代码。不是说"用 get_bytes 读取"，而是写出完整的调用和上下文
- **有可复用代码** — 对话中出现的脚本、工具调用、分析代码要保留下来，格式化好，加上注释，让学生以后能直接用
- **有具体数据** — 地址、字节序列、函数名、输出结果，这些细节要保留，它们是知识的锚点
- **有 AI 的思考过程** — 在关键节点还原 AI 是怎么推理的：看到了什么线索、排除了什么方向、为什么选择了这条路。用"当时我在想..."或"这里的推理链条是..."来引出
- **有类比和直觉** — 把抽象概念用具体事物来比喻
- **预测困惑点** — "你可能会觉得奇怪，为什么...实际上是因为..."
- **有态度有观点** — "这个细节很重要，很多新手会忽略"
- **用"我们"和"你"** — "让我们来看看..."、"你需要注意的是..."

**实操细节的要求（关键！）：**

| 情况 | 要写什么 |
|------|----------|
| 用了某个工具/API | 写出完整的调用方式、参数含义、返回值怎么看 |
| 写了分析脚本 | 把脚本完整保留，加注释说明每步在做什么 |
| 有具体的地址/字节 | 保留地址、字节序列、函数名——这是知识的锚点 |
| 有操作步骤 | 写成 1-2-3 步，每步写清楚"做什么"和"为什么" |
| 有踩坑经验 | 写清楚错误路径是什么、为什么错、正确做法是什么 |
| AI 做出了关键判断 | 还原推理链条：看到了 X → 排除了 Y → 因为 Z → 所以选择了 W |
| AI 走了弯路又回头 | 写清楚：先试了什么、为什么不work、什么线索让 AI 调整方向 |

**怎么写思考过程：**

不要写成事后诸葛亮的总结，而是还原当时的心路历程。用 `> 💭 思考过程` 引用块来标注，让学生一眼就能区分"知识"和"思考"，对用户多次提到的问题进行更加详细的教学：

```markdown
> 💡 **思考过程：** 当时我看到了 AES 解密输出全是乱码。第一反应是密钥错了——但密钥 `0xEFCDAB8967452301` 明明匹配 AES 密钥扩展的初始常量，不像是随机值。那问题不在密钥，就在加密流程本身。于是我回去仔细看加密函数 `sub_140001230` 的完整逻辑，果然在 AES 调用之前发现了一个 XOR 循环。这个推理的关键转折点是：**密钥太"整齐"了，不像是错的**——这让我把注意力从"找密钥"转向了"找额外的处理层"。
```

**绝对不要：**
- 写练习题
- 列干巴巴的"是什么/为什么/怎么做"清单
- 只提工具名不给具体用法（"用 get_bytes" → 要写出怎么调用）
- 把代码丢掉只留概述（"写了个脚本读取..." → 要贴出完整脚本）
- 用 AI 冰冷的第三人称叙述

### 第四步：保存到文档

**输出目录：** `E:\AI\SKILL\learn\`

**文件命名：** `{YYYY-MM-DD}_{主题缩写}.md`

如果当天该主题的文件已存在，**追加内容**而不是覆盖。

在保存到文档时，根据文档模板在tags内添加标签，这个标签是用来说明当前笔记里涉及到了哪些知识点，tags数量至少有一个，最多10个。并扫描输出目录中所有md文件最前端以`---`包裹的几行内容，查看tags里是否有与该文档相关的内容，如果有就在links中添加文件名称，以obsidian的格式进行链接。

## 快速参考

| 场景 | 操作 |
|------|------|
| 上下文即将压缩 | 全量扫描——提取所有知识点 |
| `/learning-journal` 命令 | 全量扫描 + 详细输出 |
| 一次会话多个领域 | 每个领域一个章节 |
| 知识点很少时 | 合并为一个小节，不必拆分 |
| 工具用法 | 单独一个"工具实操手册"章节，给具体示例 |

## 上下文压缩协议

当上下文即将被压缩时，在压缩之前执行：

1. **扫描** — 回顾完整对话，识别所有知识点
2. **归类** — 按知识领域分组
3. **讲解** — 以老师的口吻把每个知识点讲清楚，带上实操细节和可复用代码
4. **写入** — 生成学习文档
5. **确认** — 告诉人类伙伴保存了什么、保存在哪里

## 常见错误

| 错误 | 修正 |
|------|------|
| 以问题为主线组织文档 | 以知识领域为主线 |
| 用"是什么/为什么/怎么做"割裂知识 | 像讲课一样连贯地讲 |
| 写练习题 | 不要写，只讲知识 |
| 只提工具名不给具体用法 | 写出完整的调用方式和参数 |
| 把代码丢掉只留概述 | 保留完整脚本，加注释 |
| 丢掉地址、字节序列等细节 | 这些是知识的锚点，要保留 |
| 老师口吻变成 AI 冰冷叙述 | 用"让我们来"、"你需要"、"这个很关键" |
| 堆砌术语不加解释 | 每个术语第一次出现时都要解释 |

## 高价值信号——立刻捕获

- 对话中解释了某个概念的工作原理
- 提及了某个技术的基础理论
- 讨论了为什么某种做法是最佳实践
- 对比了不同技术方案的取舍
- 涉及了某个领域的专业术语和背景知识
- 出现了可复用的分析脚本或工具调用方式
